# 智慧萌宠护送平台答辩 100 问

> 项目名称：智慧萌宠护送 IoT 平台  
> 项目定位：面向宠物主人、调度员、司机、照护员和管理员的宠物运输可视化与智能协同平台  
> 使用建议：先掌握每题第一句话，再根据老师追问补充后面的技术细节。回答时必须区分“已实现功能”“演示数据”和“后续规划”。

## 一、产品定位与业务设计（1—10）

### 1. 你们的项目解决了什么问题？

**参考回答：** 我们解决的是宠物异地运输过程中“主人看不见、环境不可知、异常响应慢”的问题。普通物流只关心车辆和货物是否到达，而宠物运输还需要持续关注温湿度、空气质量、震动、照护记录、预计到达时间和动物福利风险，因此我们把传统物流跟踪升级成了宠物旅程管理。

### 2. 为什么选择宠物护送，而不是普通物流？

**参考回答：** 宠物不是普通货物，运输过程具有生命安全、情绪应激和照护责任等特殊要求。这个方向既能复用车辆定位、IoT 监控和调度能力，又能形成明显的垂直场景差异，体现项目从通用技术底座向真实业务产品演进的能力。

### 3. 项目的主要用户有哪些？

**参考回答：** 主要有五类用户：宠物主人、调度员、司机、照护员和系统管理员。宠物主人关注自己的宠物、位置、环境和 ETA；调度员关注全局风险和车辆调度；司机关注路线和任务；照护员负责饮水、休息、检查等记录；管理员负责账号、设备、权限和审计。

### 4. 项目的核心业务流程是什么？

**参考回答：** 核心流程是：创建宠物档案，生成运输旅程，将宠物、笼位和车辆绑定，车辆设备持续上报 GPS 与环境数据，平台计算位置、ETA 和风险，主人实时查看，异常时通知调度与照护人员，最终完成签收并生成运输报告。

### 5. 货主端为什么要单独设计一套界面？

**参考回答：** 因为货主也就是宠物主人，不需要看到车辆管理、人员管理和全平台告警等复杂后台功能。他们最关心的是“我的宠物是否安全、现在在哪里、什么时候到、有没有被照护”，所以货主端采用中心助手加少量关键指标的低认知负担设计。

### 6. 货主端有哪些权限边界？

**参考回答：** 货主只能查询自己名下宠物的旅程、环境、ETA、照护记录和与自己相关的告警，不能查看其他用户数据，也不能直接调度车辆、修改路线、关闭告警或管理人员。即使 Agent 理解到了调度类意图，前端也不会直接执行写操作。

### 7. 项目的主要创新点是什么？

**参考回答：** 第一是把 IoT 物流监控转换成动物福利监控；第二是用“羊小智”统一承载语音问答、状态解释和风险提醒；第三是将 Agent 的自然语言能力和前端动作执行结合，同时通过角色权限和二次确认控制风险；第四是支持真实 API 与 mock 演示双模式。

### 8. “羊小智”在产品中起什么作用？

**参考回答：** 羊小智不是纯装饰，而是宠物运输助手的视觉人格。它负责接收语音或文字问题、解释运输数据、播报告警和安抚主人焦虑。角色文案明确不冒充兽医、不做疾病诊断，也不隐藏高风险事件。

### 9. 为什么首页采用中心式布局？

**参考回答：** 中心式布局让宠物助手成为第一视觉焦点，周围六个入口对应追踪、告警、运输、宠物档案、分析和设置，左右两侧放高频状态信息。这样既接近用户提供的原型，也符合宠物主人“先确认安全，再看细节”的浏览顺序。

### 10. 当前产品还有哪些不足？

**参考回答：** 当前宠物数据主要基于物流数据模型改造，部分后台页面仍保留原物流语义；真实 TTS、ASR、Kafka、MQTT 和对象存储依赖外部服务授权与运行环境；宠物健康评估目前是规则提醒而不是医学诊断。后续会继续补充宠物专属表结构、真实设备数据和完整照护闭环。

## 二、前端技术栈与架构（11—30）

### 11. 前端使用了什么技术栈？

**参考回答：** 前端使用 Vue 3.5、TypeScript 5.7、Vite 6、Vue Router 4、Pinia 2、Element Plus 2.9、Axios、ECharts、Leaflet、高德地图 JS API、MQTT.js 和原生 WebSocket。Vue 负责组件化，Pinia 管理业务状态，Axios 统一 HTTP 请求，地图和图表负责可视化。

### 12. 为什么选择 Vue 3？

**参考回答：** Vue 3 的 Composition API 适合组织录音、实时连接、地图和复杂状态等逻辑，响应式系统也适合实时数据更新。与 TypeScript、Vite 和 Element Plus 配合后，开发效率高，组件生态成熟，适合中后台与可视化项目。

### 13. 为什么使用 TypeScript？

**参考回答：** TypeScript 可以约束车辆、货物、用户、告警、Agent Action 等数据结构，减少接口字段写错和空值问题。项目通过 `vue-tsc` 在构建前做类型检查，使很多错误在浏览器运行前就能发现。

### 14. Vite 在项目中的作用是什么？

**参考回答：** Vite 提供快速开发服务器、热更新、TypeScript 与 Vue 单文件组件构建，并通过开发代理把 `/api` 请求转发到 `localhost:8080`，解决本地联调中的跨域问题。生产构建还按 Vue、Element Plus、Leaflet 等依赖进行分包。

### 15. 前端路由是如何设计的？

**参考回答：** 项目使用 Hash 路由，包含登录、导航门户、货主工作台、总览、追踪、调度、人员、司机、告警、仓库和助手等页面。路由守卫先检查登录状态，再读取角色并校验 `meta.roles`，无权限时返回门户页。

### 16. 为什么使用 Hash 路由？

**参考回答：** Hash 路由部署简单，刷新子页面时不要求 Nginx 额外配置 history fallback，适合课程项目和静态资源部署。代价是 URL 中包含 `#`，正式产品可以改为 history 模式并配置服务器回退。

### 17. Pinia 管理了哪些状态？

**参考回答：** Pinia 主要管理当前用户、角色、车辆、宠物旅程的兼容数据、告警、命令、通知、实时连接状态、头像和演示数据。页面只消费 Store 暴露的数据与方法，避免每个组件重复请求和维护相互冲突的状态。

### 18. API 请求如何统一处理？

**参考回答：** `services/http.ts` 基于 Axios 创建统一客户端，配置基础地址、超时、请求头、Token 注入、错误解析和 Token 刷新。`services/api.ts` 再按认证、车辆、货物、告警、助手等业务域封装具体接口。

### 19. 前端怎样处理登录状态？

**参考回答：** Token 管理器统一保存 access token、refresh token、过期时间和用户信息。路由守卫检查 Token，Axios 请求拦截器自动携带 Bearer Token，发现即将过期时先刷新，刷新失败则清理会话并跳回登录页。

### 20. 为什么项目支持 mock 和 API 两种模式？

**参考回答：** mock 模式保证没有后端或外部服务时仍能完整演示页面与交互；API 模式连接真实后端与实时数据。模式由环境变量控制，开发、答辩和正式联调可以使用不同配置，不需要修改组件代码。

### 21. 货主工作台是怎样实现的？

**参考回答：** 货主工作台是独立的 `PetOwnerDashboardView.vue`，包含品牌头部、运输概览、宠物卡片、最近告警、运输状态、环境监测、中心助手和六个功能入口。它不复用深色调度后台的外壳，因此能保持面向消费者的轻量视觉风格。

### 22. 语音对话为什么不再跳转页面？

**参考回答：** 用户点击中心“点击对话”后，通过组件暴露的 `openAndListen()` 打开当前页面内的对话层并立即申请麦克风。识别文字、Agent 回复、语音播放都在同一对话层完成，降低上下文切换，也更符合主人和助手连续交谈的体验。

### 23. 浏览器语音识别为什么会自动结束？你们怎么修复？

**参考回答：** 浏览器 `SpeechRecognition` 在短暂无声或一段结果结束时会触发 `onend`。项目增加了持续监听状态：无结果时自动重启，出现最终文本后等待约 1.8 秒静音再发送，最长监听 12 秒，再次点击麦克风也可以手动结束。

### 24. 前端语音如何兼容服务不可用？

**参考回答：** Agent 接口不可用时使用货主只读 mock 规则回答；豆包 TTS 不可用时使用浏览器 `speechSynthesis`，优先选择中文女声并设置较高音调和适中语速。这样外部 API 故障不会让答辩演示中断。

### 25. 为什么使用 Element Plus？

**参考回答：** Element Plus 提供表单、弹窗、消息、抽屉和图标等成熟组件，适合业务后台并降低重复开发成本。宠物主人首页采用自定义 CSS 保持独特视觉，但确认框、错误反馈和基础控件仍复用 Element Plus。

### 26. 地图为什么同时支持高德和 Leaflet？

**参考回答：** 高德地图适合国内真实地址、路线和坐标服务；Leaflet 轻量且可使用自定义底图，适合离线演示和可控 mock 场景。项目封装 `SmartMap` 与 `LeafletFleetMap`，页面不直接依赖单一地图实现。

### 27. 前端如何接收实时数据？

**参考回答：** `services/websocket.ts` 负责后端 WebSocket 事件，`services/mqtt.ts` 可通过 WebSocket MQTT 连接 Broker。Store 订阅位置、心跳、告警和指令事件后更新响应式数据，页面和地图会自动重新渲染。

### 28. 如何避免实时连接无限重连？

**参考回答：** WebSocket 客户端采用递增退避策略，重连间隔从较小值逐步增加并设置上限；同时记录连接状态，在页面提示“实时连接正常”或“恢复中”。还应在退出登录和组件销毁时主动关闭连接与定时器。

### 29. 前端如何做响应式布局？

**参考回答：** 桌面端使用三列网格，中心是助手舞台，左右是状态卡；中等屏幕改为中心加侧栏；移动端改为纵向排列，并让对话层变成底部弹层。固定格式元素使用明确尺寸和断点，避免文字或圆形入口互相覆盖。

### 30. 前端如何优化性能？

**参考回答：** 页面组件使用异步加载，地图延迟预加载，Vite 构建按大型依赖分包；实时事件只更新必要状态，避免整页刷新；图片使用合理尺寸和透明素材；组件卸载时清理监听器、定时器、音频对象 URL 和媒体流。

## 三、后端架构与接口设计（31—50）

### 31. 后端使用了什么技术栈？

**参考回答：** 后端基于 Java 17 和 Spring Boot 2.7.3，使用 Spring MVC、MyBatis、PostgreSQL 驱动、Druid、Redis、Spring Kafka、Spring Integration MQTT、WebSocket、MinIO、pgvector、Knife4j 和 Lombok。

### 32. 后端为什么采用多模块 Maven？

**参考回答：** `sky-common` 放公共结果、异常和工具，`sky-server` 放业务 Controller、Service、Mapper 和配置。多模块能隔离公共能力与业务服务，减少重复依赖，也便于以后拆分独立服务。

### 33. 后端采用什么分层结构？

**参考回答：** 典型结构是 Controller 接收和校验请求，Service 实现业务规则，Mapper 访问数据库，DTO 承接输入参数，Entity 对应持久化对象，VO 对外返回页面所需数据。统一异常处理器将异常转换成标准 API 响应。

### 34. 为什么不直接返回数据库 Entity？

**参考回答：** Entity 与数据库结构绑定，直接返回会泄露内部字段并造成接口与表结构强耦合。DTO 和 VO 可以分别约束输入与输出，例如人脸绑定接口使用 `FaceBinding` 实体读取数据库，再转换成 `FaceBindingVO` 返回。

### 35. API 响应格式是什么？

**参考回答：** 业务接口使用统一 `ApiResponse<T>`，包含业务 code、message、data、timestamp 和 requestId 等信息。前端只在 code 为成功值时返回 data，否则转成统一 `ApiError`，便于全局处理。

### 36. 后端怎样进行异常处理？

**参考回答：** `LogisticsExceptionHandler` 使用全局异常处理，将认证失败、权限不足、业务异常和未知异常映射为不同错误码与 HTTP 状态。这样 Controller 不需要重复写 try-catch，前端也能得到一致错误结构。

### 37. 登录接口的基本流程是什么？

**参考回答：** 前端提交账号密码，后端查询用户并验证状态与凭证，生成 access token 和 refresh token，返回用户角色与权限。后续请求由安全服务解析 Token，形成当前用户上下文，再执行数据范围校验。

### 38. 为什么需要 refresh token？

**参考回答：** access token 生命周期较短，泄漏后的风险较低；refresh token 用于无感续期。前端在 access token 快过期时调用刷新接口，刷新失败才要求重新登录，兼顾安全性和用户体验。

### 39. 如何实现角色权限？

**参考回答：** 前端路由的 `meta.roles` 负责界面级拦截，后端 `LogisticsSecurityService` 和用户上下文负责最终授权。前端限制不能替代后端限制，因为用户可以绕过页面直接请求 API。

### 40. 如何保证货主只能看到自己的宠物？

**参考回答：** 正式实现应从 Token 获取当前用户 ID，查询旅程时强制增加 `owner_id = currentUserId` 条件，而不是相信前端传入 ownerId。详情、轨迹、告警和照护记录接口都要执行同样的数据归属检查。

### 41. Controller 为什么使用 REST 风格？

**参考回答：** REST 风格用资源路径和 HTTP 方法表达语义，例如 `/vehicles`、`/cargo/{id}`、`/alerts/{id}/acknowledge`。路径可读，前后端职责清晰，也便于 Swagger/Knife4j 自动生成文档。

### 42. 后端怎样实现分页？

**参考回答：** 查询 DTO 接收 page、size、状态和关键词，Mapper 执行条件查询，Service 返回统一分页对象，包含 content、总数和页码。分页避免一次返回大量车辆、告警或日志数据。

### 43. MyBatis 的作用是什么？

**参考回答：** MyBatis 负责 SQL 与 Java 对象映射，项目通过 Mapper 接口和 XML 控制复杂查询。相比全自动 ORM，它对物流中的条件查询、统计、时序关联和显式字段映射更可控。

### 44. 为什么使用 Druid 数据源？

**参考回答：** Druid 提供连接池、连接复用和监控能力，减少频繁创建数据库连接的成本。连接参数由环境变量配置，不同环境可以切换地址和密码。

### 45. Knife4j 用来做什么？

**参考回答：** Knife4j 基于 Swagger 注解生成交互式 API 文档，方便查看接口路径、参数、响应和在线调试。答辩时可以通过它展示后端并非只有页面，而是有清晰的接口能力。

### 46. 后端如何调用第三方 API？

**参考回答：** 项目使用带连接和读取超时的 `RestTemplate` 调用高德、百度、DeepSeek、SiliconFlow 和豆包等服务。第三方异常会被捕获并转换为可理解的业务错误，避免原始堆栈直接暴露给前端。

### 47. 为什么第三方密钥不能写在前端？

**参考回答：** 前端代码会下载到用户浏览器，任何人都能查看源码和网络请求，因此密钥必须放在后端环境变量或本机忽略文件中。前端只调用自己的后端接口，由后端加密钥请求第三方服务。

### 48. `.env.properties` 的作用是什么？

**参考回答：** Spring Boot 通过 `spring.config.import` 可选加载本机 `.env.properties`，其中保存豆包、百度等本地密钥。该文件加入 `.gitignore`，不会提交到仓库，生产环境则使用系统环境变量或密钥管理服务。

### 49. 后端启动时部分中间件不可用怎么办？

**参考回答：** 对非核心依赖应采用可降级启动，例如 MinIO 初始化失败只记录警告；Kafka、MQTT 可以通过环境变量关闭自动启动；核心数据库不可用则应明确启动失败。课程演示可以关闭暂时不用的消费者，降低日志噪声。

### 50. 你们如何保证接口可扩展？

**参考回答：** 按业务域划分 Controller 和 Service，使用 DTO/VO 稳定接口契约，第三方能力通过独立 Service 封装，配置通过环境变量注入。以后替换 ASR、TTS 或地图供应商时，页面和核心业务不需要整体重写。

## 四、数据库、实时通信与 IoT（51—65）

### 51. 为什么同时使用 PostgreSQL 和 TimescaleDB？

**参考回答：** PostgreSQL 保存用户、车辆、宠物旅程、绑定、告警和操作日志等强关系业务数据；TimescaleDB 是 PostgreSQL 的时序扩展，适合保存高频 GPS 点和设备心跳。两者分工后，业务事务与时序写入互不干扰。

### 52. 哪些数据适合放在 TimescaleDB？

**参考回答：** 带时间戳、持续高速写入、常按时间范围查询的数据适合放在 TimescaleDB，例如经纬度、速度、方向、温湿度、震动、设备在线心跳。宠物档案和订单状态不属于高频时序数据，应放普通业务库。

### 53. Redis 在项目中的作用是什么？

**参考回答：** Redis 可用于缓存车辆最新位置、设备在线状态、热点查询、Token 或短期幂等标记，降低数据库压力。缓存数据必须设置合理过期时间，不能把 Redis 当唯一持久化数据库。

### 54. MQTT 为什么适合 IoT 设备？

**参考回答：** MQTT 协议轻量、报文小、支持发布订阅和不同 QoS，适合网络不稳定的车载设备。车辆向 `vehicle/{id}/gps` 和 heartbeat 等主题发布数据，后端统一订阅，不需要每台设备和业务服务建立强耦合连接。

### 55. Kafka 在系统中承担什么角色？

**参考回答：** Kafka 用于把实时采集和后续处理解耦。MQTT 接收到 GPS 或心跳后可以写入 Kafka，轨迹存储、偏航检测、在线状态和告警服务分别消费，使高峰数据不会直接压垮业务接口。

### 56. MQTT 和 Kafka 有什么区别？

**参考回答：** MQTT 更靠近设备接入，强调轻量发布订阅；Kafka 更靠近服务端数据管道，强调高吞吐、持久化日志和消费者组。项目中 MQTT 解决“设备怎么上报”，Kafka 解决“服务怎么可靠处理”。

### 57. WebSocket 的用途是什么？

**参考回答：** WebSocket 用于把后端状态变化主动推送到浏览器，例如车辆位置、设备上下线、告警和调度指令。相比前端每几秒轮询，WebSocket 延迟更低、无效请求更少。

### 58. 如何判断设备在线或离线？

**参考回答：** 后端记录设备最后心跳时间，定时任务比较当前时间和阈值；超过阈值则标记离线并生成事件，恢复心跳后再标记在线。必须增加防抖，避免短暂网络抖动造成频繁上下线告警。

### 59. GPS 数据如何从设备到达前端？

**参考回答：** 典型链路是：设备通过 MQTT 上报，接收服务解析并写入 Kafka，消费者存入 TimescaleDB并更新最新位置缓存，WebSocket 将增量位置推送到前端，地图根据车辆 ID 更新 Marker。

### 60. 如何做偏航检测？

**参考回答：** 后端将最新 GPS 点与规划路线进行距离计算，当偏离距离超过阈值并持续一定时间才生成告警。这样能过滤定位漂移和临时变道；告警中记录当前位置、偏离距离、持续时间和建议动作。

### 61. ETA 是如何计算的？

**参考回答：** 基础 ETA 由剩余路程和当前或历史速度估算，真实模式可以调用高德路线规划获得道路级 ETA，再结合停留、偏航和交通状态修正。前端展示 ETA 时同时展示更新时间，避免把旧值当实时结果。

### 62. 环境监测有哪些指标？

**参考回答：** 当前产品关注温度、湿度、空气质量、震动和光照，后续可扩展氧气、噪声、笼门状态、饮水和摄像头在线状态。每种宠物和笼位可以配置不同安全阈值。

### 63. 告警等级如何设计？

**参考回答：** 可分为 INFO、LOW、MEDIUM、HIGH 和 CRITICAL。严重风险立即打断并持续提示，中风险进入通知列表，恢复后发送恢复事件；同一宠物同一风险在时间窗口内去重，避免告警轰炸。

### 64. 如何保证消息不会重复处理？

**参考回答：** 每个设备事件携带唯一 eventId，消费者在数据库或 Redis 中记录已处理 ID；写入使用唯一约束或幂等更新。Kafka 消费成功后再提交 offset，即使重复投递也不会生成重复告警或状态记录。

### 65. 如果网络断开，系统如何处理？

**参考回答：** 设备端可以本地缓存未发送数据并重连补报，后端根据心跳标记离线，前端显示连接恢复中并停止把旧位置当实时位置。恢复后按时间戳补齐轨迹，同时区分“实时点”和“补传点”。

## 五、AI、Agent、语音与知识库（66—80）

### 66. 项目中的 Agent 是什么？

**参考回答：** Agent 是把自然语言转成结构化业务动作的执行层。用户说“布丁现在到哪里了”，后端识别查询意图并返回回复或 Action；前端只执行白名单内、符合当前角色权限的动作。

### 67. Agent Action 有哪些类型？

**参考回答：** 主要包括 NAVIGATE、HIGHLIGHT_MAP_TARGET、OPEN_MODAL、SHOW_RESULT、CALL_API、FUNCTION_CALL 和 NOOP。Action 将大模型输出与具体前端行为解耦，避免模型直接操作 DOM 或随意请求接口。

### 68. 为什么需要 Action 白名单？

**参考回答：** 大模型输出具有不确定性，如果允许任意路由或 URL，会产生越权和安全风险。白名单限定可跳转页面、可调用函数和参数结构，未知 Action 直接拒绝并提示。

### 69. 哪些 Agent 操作必须二次确认？

**参考回答：** 改道、停车、关闭告警、修改地址、修改健康备注、标记交付和删除任务等写操作必须确认。查询位置、环境和 ETA 属于只读操作，可以直接执行。

### 70. 货主 Agent 与调度 Agent 有什么不同？

**参考回答：** 货主 Agent 只允许查询自己宠物的数据，并提供解释与安抚；调度 Agent 可以查看全局车辆和风险，在确认后下发调度指令。两者可以共用大模型，但系统提示词、工具列表和后端数据范围不同。

### 71. 语音交互完整链路是什么？

**参考回答：** 用户点击麦克风，浏览器采集或识别语音，得到文字后请求 Agent，Agent 返回只读回复，前端显示消息，再调用后端 TTS 生成音频并播放。任一外部服务失败时都有文字或浏览器语音降级。

### 72. 当前语音识别使用什么方案？

**参考回答：** 货主对话层优先使用 Chrome/Edge 提供的 `SpeechRecognition` 以降低演示依赖；后端同时具备上传 WAV 后调用 ASR 的接口，可配置百度 ASR 或其他识别服务。生产环境应优先使用可控的服务端 ASR。

### 73. 为什么录音需要 HTTPS？

**参考回答：** 浏览器把麦克风和摄像头视为敏感权限，除 localhost 外通常只允许安全上下文，也就是 HTTPS。部署到服务器后如果仍使用 HTTP，按钮代码正确也无法获得媒体权限。

### 74. 豆包 TTS 是如何接入的？

**参考回答：** 前端向自己的 `/api/v1/assistant/speech` 发送文本，后端 `DoubaoTtsService` 使用 `X-Api-Key` 调用豆包 HTTP 音频生成接口，模型为 `seed-audio-1.0`，传入 speaker 和音频参数，解析 Base64 音频后以 MP3 返回浏览器。

### 75. 为什么选可爱女声音色？

**参考回答：** 货主场景需要温柔、可信和陪伴感，过于严肃的播报会放大主人焦虑。项目使用可配置的中文女声音色，例如 `zh_female_vv_uranus_bigtts`，同时控制语速和音调，告警内容仍保持客观准确。

### 76. 豆包 TTS 调用还需要哪些条件？

**参考回答：** 除音色 ID 外，还需要新版语音控制台生成的 X-Api-Key、`seed-audio-1.0` 模型权限、对应资源授权、文本 Prompt 和输出音频配置。仅有一个普通方舟 Key 或未授权资源时，接口会返回 401 或 403。

### 77. TTS 失败时为什么还可以播报？

**参考回答：** 前端捕获 `/assistant/speech` 失败后调用浏览器 `speechSynthesis`，优先选择中文女声。这个降级不代表豆包调用成功，而是为了保证用户仍能听到回答，并在日志中保留真实错误供排查。

### 78. 大模型在项目中是否会直接访问数据库？

**参考回答：** 不会。大模型只选择受控工具并生成结构化参数，真正的数据查询由后端 Service 和 Mapper 完成。这样可以执行权限校验、SQL 参数化、日志记录和结果过滤。

### 79. RAG 知识库解决什么问题？

**参考回答：** RAG 用于回答运输规范、设备故障处理和照护流程等文档问题。文档上传后切分成片段，Embedding 转成向量存入 pgvector，提问时检索相关片段并交给大模型生成带来源的回答。

### 80. 如何降低大模型幻觉？

**参考回答：** 业务数据通过工具查询，不让模型凭空编造；知识问题使用 RAG 并显示来源；系统提示词规定不知道就明确说明；高风险操作必须确认；回复中的位置、温度和 ETA 应携带更新时间或数据来源。

## 六、安全、测试、部署与运维（81—95）

### 81. 系统有哪些主要安全风险？

**参考回答：** 主要风险包括密钥泄漏、越权查询、Token 被盗、Agent 执行危险动作、上传恶意文件、接口注入和日志泄露隐私。对应措施是服务端鉴权、数据范围校验、Action 白名单、参数化 SQL、文件校验、HTTPS 和敏感信息脱敏。

### 82. 为什么前端隐藏按钮不能算权限控制？

**参考回答：** 用户可以用开发者工具直接构造请求，所以隐藏菜单只能改善界面，不能保证安全。真正的权限必须在后端根据 Token、角色、资源归属和操作类型再次校验。

### 83. 如何防止 SQL 注入？

**参考回答：** MyBatis 使用参数绑定而不是字符串拼接，动态排序字段使用后端白名单，模糊查询也通过参数传递。所有来自前端和大模型的参数都视为不可信输入。

### 84. 文件上传如何保证安全？

**参考回答：** 后端限制扩展名、MIME、大小和数量，生成服务器文件名而不使用用户原名，文件存储在非执行目录或 MinIO，对外只返回受控访问地址。知识库目前主要允许 Markdown 和文本类型。

### 85. 人脸识别图片存在哪里？

**参考回答：** 标准图片元数据和本地业务绑定保存在系统数据库，图片可存本地受控上传目录或对象存储；百度负责识别特征和搜索，不应把百度当作唯一业务档案库。删除绑定时要同步处理本地记录和百度用户。

### 86. 人脸登录如何避免误识别？

**参考回答：** 设置相似度阈值、限制用户组、增加活体检测和失败次数限制，并记录设备与登录日志。高安全场景可以要求人脸加短信或密码，而不是只依赖单一生物特征。

### 87. 前端怎样防止 XSS？

**参考回答：** Vue 默认会转义插值文本；Markdown 或富文本渲染前使用 DOMPurify 清洗；不对用户输入使用未经处理的 `v-html`。后端也应对输出和上传内容做验证。

### 88. 项目做了哪些构建验证？

**参考回答：** 前端执行 `vue-tsc -b && vite build`，同时完成类型检查和生产构建；后端执行 `mvn -DskipTests package` 验证所有模块可以编译并生成可执行 JAR。关键页面还通过桌面和窄屏截图检查布局。

### 89. 应该怎样测试语音功能？

**参考回答：** 测试麦克风允许、拒绝、无设备、短暂无声、连续说话、手动结束、12 秒超时、ASR 失败、Agent 失败、TTS 失败和浏览器降级。还要验证录音结束后媒体轨道与定时器被释放。

### 90. 应该怎样测试角色权限？

**参考回答：** 分别用五种角色登录，验证菜单、路由、API 和数据范围；再绕过前端直接请求无权限接口，确认后端返回 403。货主还要用其他用户的宠物 ID 测试水平越权。

### 91. 应该怎样测试实时数据？

**参考回答：** 模拟正常 GPS、重复事件、乱序事件、断网补传、心跳超时和异常恢复；检查数据库写入、缓存、告警、WebSocket 推送和前端 Marker 是否一致，并验证重复事件不会重复告警。

### 92. 本地如何启动项目？

**参考回答：** 前端进入 `smart-logistics-ui` 执行 `npm run dev -- --port 5173`；后端进入 `sky-take-out` 执行 `mvn spring-boot:run -pl sky-server`，或运行构建后的 JAR。后端默认端口 8080，Vite 将 `/api` 代理到该端口。

### 93. 生产环境怎样部署？

**参考回答：** 前端执行生产构建后由 Nginx 提供静态资源，`/api` 和 WebSocket 反向代理到 Spring Boot；后端用 systemd 或容器运行；数据库和中间件独立部署；域名配置 HTTPS，密钥通过环境变量注入。

### 94. 为什么需要 Nginx？

**参考回答：** Nginx 负责 HTTPS、静态资源、缓存、压缩、API 反向代理和 WebSocket Upgrade，也可以统一前后端域名避免 CORS。生产环境不应直接把 Vite 开发服务器暴露给用户。

### 95. 系统怎样做可观测性？

**参考回答：** 每个请求生成 requestId，外部 API 保存 traceId 或 logId；日志区分 INFO、WARN、ERROR；监控接口响应时间、WebSocket 连接数、设备在线率、消息积压和告警数量。正式环境可接入 Prometheus、Grafana 和集中日志系统。

### 96. 你在项目中负责什么？

### 97. 项目中最难的问题是什么？

**参考回答：** 最难的是把语音、Agent、实时数据和角色权限组合成稳定闭环。语音识别会自动结束，外部 TTS 可能未授权，大模型动作存在不确定性，所以我们设计了持续监听、只读权限、Action 白名单、二次确认和多级降级。

### 99. 这个项目怎样证明不是只有 UI？

**参考回答：** 可以现场展示 Spring Boot 接口、数据库、WebSocket、MQTT/Kafka 配置、Agent Action、语音上传与 TTS Service，再用前端触发请求观察后端日志。页面只是交互入口，背后有业务分层、数据存储、实时链路和第三方服务封装。

### 100. 请用一分钟总结项目。

**参考回答：** 我们将传统智慧物流平台升级为智慧萌宠护送平台，核心目标是让主人实时知道宠物在哪里、环境是否安全、是否得到照护以及何时到达。系统前端使用 Vue 3、TypeScript、Pinia 和地图可视化，后端使用 Spring Boot、MyBatis、PostgreSQL 与 TimescaleDB，并通过 MQTT、Kafka 和 WebSocket形成实时链路。项目还引入语音 Agent 和豆包 TTS，让主人可以直接询问宠物状态，同时使用角色权限、白名单和降级机制保证安全与可演示性。当前已经完成货主工作台、语音对话、核心物流接口和实时技术底座，下一步将完善宠物专属数据模型和真实设备闭环。

### D.3 后端返回的权限标识

| 角色 | 主要权限标识 | 权限含义 |
|---|---|---|
| `SHIPPER` | `CARGO_READ`、`POSITION_READ`、`ASSISTANT_CHAT` | 查询自己的运输任务、位置与 ETA，使用货主助手 |
| `WAREHOUSE` | `VEHICLE_WRITE`、`CARGO_WRITE`、`BINDING_WRITE` | 维护入库运输记录、车辆绑定和仓储状态 |
| `DISPATCHER` | `VEHICLE_READ`、`COMMAND_SEND`、`ALERT_READ` | 查看车队、下发调度指令、处理运输告警 |
| `DRIVER` | `TASK_READ`、`STATUS_REPORT` | 查看本人任务并上报执行状态 |
| `ADMIN` | `SYSTEM_ADMIN`、`VEHICLE_WRITE`、`CARGO_WRITE`、`ALERT_WRITE` | 管理用户、角色和全局业务资源 |

### D.4 权限如何真正生效

1. **菜单层：** `App.vue` 根据当前用户角色过滤菜单，减少无权限入口。
2. **路由层：** `router.ts` 检查登录状态和 `meta.roles`，手动输入无权地址会返回导航门户。
3. **认证层：** 后端拦截 `/api/v1/**` 和兼容 `/api/**`，校验 JWT；登录、人脸登录、刷新 Token 和公开图片读取除外。
4. **接口层：** 车辆、货物、告警、用户等敏感写操作再次检查角色，不能依靠前端隐藏按钮保证安全。
5. **账号层：** 普通用户只能修改本人资料和本人人脸，不能修改自身角色；只有管理员可以管理用户和分配角色。

答辩时可以概括为：**前端菜单控制可见性，路由守卫防止误入，后端 JWT 确认身份，接口角色校验决定是否真正允许操作。**
