## 3. 顶层目标

平台面向宠物主人、仓管、调度员、司机和管理员，解决“宠物运单、车辆、设备、调度、环境风险、智能交互”的统一协作问题。技术底座继续复用原智慧物流实时链路，但产品中心从普通货物调整为有生命运输对象，增加宠物状态可见、异常及时触达、交接可追溯和最小权限访问要求。

顶层设计围绕六个闭环：

| 闭环 | 目标 |
|---|---|
| 护送受理闭环 | 货主提交需求，仓管核验并交接，调度员分配车辆和司机。 |
| 运输可视化闭环 | GPS/环境数据上报后，授权用户看到车辆位置、轨迹、ETA、设备和宠物运单状态。 |
| 调度指令闭环 | 调度员下发改道/通知，后端落库并通过 MQTT 发到车机，回执后前端更新。 |
| 告警处理闭环 | 离线、偏航、环境等异常被检测、去重、推送、确认、关闭并留痕。 |
| 智能交互闭环 | 用户通过语音或文本生成结构化 action，前端执行跳转、定位、弹窗或调用接口。 |
| 交付评价闭环 | 司机完成交接，货主确认收货和评价，运力释放且全程可追溯。 |

## 4. 分层设计

### 4.1 用户层

用户层覆盖货主、调度员、仓管、司机、管理员。不同角色看到不同菜单，但统一使用同一个 Vue 前端：

| 角色 | 主要页面 |
|---|---|
| 货主 `SHIPPER` | 萌宠护送首页、本人运单追踪、改址/收货/评价、智能问答 |
| 调度员 `DISPATCHER` | 车辆调度、运单追踪、告警协调、人员运营、改址审核 |
| 仓管 `WAREHOUSE` | 接宠交接、装载绑定、设备在线、环境/仓储告警处理 |
| 司机 `DRIVER` | 本人任务、状态上报、调度指令执行与回执 |
| 管理员 `ADMIN` | 用户角色、全局配置、审计和故障排查 |

角色页面和操作矩阵见[业务与权限设计第 4、5 节](./萌宠护送业务与权限设计.md#4-页面权限矩阵)。前端负责减少误操作，后端负责最终鉴权；管理员拥有治理能力，但不代替调度员和司机完成常规运输。

### 4.2 接入层

接入层由 Nginx 和 HTTP/WebSocket 入口组成：

- Nginx 承载前端静态资源、反向代理 `/api/v1` 和 WebSocket。
- HTTP API 使用统一响应格式：`code/message/data/timestamp/requestId`。
- WebSocket 统一推送车辆位置、告警、指令回执等实时事件。
- 登录态使用 JWT；前端保存 accessToken/refreshToken，后端拦截器统一拒绝匿名请求，Controller/Service 再做角色和本人权限判断。

### 4.3 前端应用层

前端主项目是 `pet-safe-transit/smart-logistics-ui/`：

| 模块 | 职责 |
|---|---|
| `src/services/*` | 统一封装 Axios、token、接口类型、WebSocket、MQTT 客户端。 |
| `src/stores/logistics.ts` | 全局业务状态：车辆、货物、告警、设备、用户、通知。 |
| `src/views/*` | 运营总览、货物追踪、车辆调度、仓储、告警、司机任务、智能问答。 |
| `src/components/SmartMap.vue` | 地图呈现与车辆定位，监听 Agent 地图定位事件。 |
| `src/components/VoiceAssistant` | 录音、上传 `/voice/command`、执行后端 action。 |
| `src/utils/actionExecutor.ts` | 前端 Action 执行器，负责跳转、地图定位、确认弹窗、调用接口。 |
| `src/views/PetOwnerDashboardView.vue` | 货主专属萌宠护送首页、运输状态、环境摘要和直接语音对话入口。 |

前端执行原则：

- 后端 Agent 只返回结构化 action，不直接操控页面。
- 页面跳转、地图聚焦、弹窗和确认后的业务 API 调用都由前端执行。
- 危险操作必须二次确认，例如下发改道指令。
- 语音入口不提升权限；语音触发的路由和接口仍经过路由守卫与后端鉴权。

### 4.4 应用服务层

后端主项目是 `pet-safe-transit/sky-take-out/sky-server/`，当前采用 Spring Boot 2.7.3 模块化单体：

| 模块 | 主要接口/职责 |
|---|---|
| 认证与用户 | `/auth/login`、`/auth/face-login`、`/users/me`、人脸绑定。 |
| 车辆与萌宠运单 | `/vehicles`、`/cargo`、车辆运单绑定、状态更新。代码保留 `cargo` 命名以兼容现有接口。 |
| 路线与地图 | `/routes/*`、`/amap/*`、高德路线规划、地理编码、偏航检测。 |
| 告警中心 | `/alerts`、告警生成、确认、关闭、日志。 |
| 调度指令 | `/vehicles/{plate}/command`、指令落库、MQTT 发布、回执处理。 |
| 设备在线 | `/devices/status`、心跳缓存、离线检测。 |
| 运单扩展 | `/orders/*`、改址影响、审核、卸货点确认、司机评价。 |
| 智能问答 | `/assistant/chat`、知识库文档、RAG、DeepSeek 调用。 |
| 语音 Agent | `/voice/command`、`/agent/command`、百度 ASR、意图解析、action 生成、豆包 TTS。 |
| 安全控制 | JWT 全局拦截、角色白名单、本人或管理员校验、关键操作日志。 |

### 4.5 实时与设备链路

实时链路的主路径是：

```txt
车载终端
  → MQTT: vehicle/{vin}/gps 或 vehicle/{vin}/heartbeat
  → EMQX
  → Spring Integration MQTT Consumer
  → Kafka: gps-points / vehicle-heartbeats
  → Kafka Consumer
  → TimescaleDB 保存历史点位
  → Redis 保存最新位置/心跳
  → WebSocket 推送给前端
```

调度指令的反向路径是：

```txt
前端确认调度指令
  → POST /vehicles/{plate}/command
  → commands / command_logs 落库
  → MQTT: vehicle/{vin}/command
  → 车载终端执行
  → command/ack 回传
  → 前端 WebSocket 更新执行状态
```

### 4.6 数据层

| 组件 | 当前用途 |
|---|---|
| PostgreSQL | 用户、车辆、萌宠运单、绑定、告警、指令、改址、人脸、评价和语音日志等业务数据。 |
| TimescaleDB | GPS 点位、设备心跳等时序数据。 |
| Redis | 最新位置、最新心跳、告警去重、token 黑名单、短期状态缓存。 |
| Kafka | GPS、心跳、告警、指令等事件缓冲和解耦。 |
| MinIO | 宠物/运单附件、知识库文档、图片等对象文件。 |
| pgvector | 知识库 chunk embedding，用于 RAG 检索。 |

## 5. 核心链路设计

### 5.0 一次萌宠护送主链路

```txt
货主提交护送需求
  → 仓管接宠、核验、装载并检查设备
  → 调度员分配车辆/司机并规划路线
  → 司机确认任务后发车
  → GPS/心跳/环境数据实时上报
  → 货主看本人运单，调度员监控全局，仓管处理设备/环境问题
  → 异常触发告警，调度员下发指令，司机执行回执
  → 到达交接，货主确认收货并评价
  → 运单归档、车辆释放、状态/轨迹/告警/指令留痕
```

正常流程、异常流程、改址协作和状态机见[萌宠护送业务与权限设计](./萌宠护送业务与权限设计.md#6-一次护送的角色协作流程)。

### 5.1 登录与人脸登录

```txt
账号密码登录：
前端 → POST /auth/login → JWT + 用户信息 → 初始化业务数据

人脸登录：
摄像头截图 → imageBase64 → POST /auth/face-login
  → 百度人脸检索/本地演示兜底
  → face_login_logs 记录
  → JWT + 用户信息 + face 匹配信息
```

设计边界：

- 前端负责摄像头采集和 base64 编码。
- 后端负责人脸比对、用户映射、登录日志和 token 签发。
- 百度 Key 未配置时使用演示兜底，只用于本地联调。

### 5.2 语音 Agent Action

```txt
点击 VOICE 按钮
  → 浏览器录音并转 16k mono WAV
  → POST /voice/command
  → 百度 ASR 转文本
  → Agent 生成 action
  → 前端 executeAgentAction(action)
  → 需要语音回复时由豆包 TTS 合成音频
```

Action 类型：

| 类型 | 前端行为 |
|---|---|
| `NAVIGATE` | 检查路由白名单后跳转页面。 |
| `HIGHLIGHT_MAP_TARGET` | 派发地图事件并定位车辆。 |
| `OPEN_MODAL` | 打开指定弹窗或帮助窗口。 |
| `SHOW_RESULT` | 展示查询结果或跳转到对应业务视图。 |
| `CALL_API` | 先弹确认框，再调用后端白名单接口。 |
| `NOOP` | 提示暂无可执行操作。 |

安全约束：

- 路由跳转必须命中 `routeWhiteList`。
- `CALL_API` 必须二次确认。
- 后端现已具备角色校验；`CALL_API` 仍需进一步收紧为接口、方法和参数三级白名单。
- ASR 低置信度、多车辆或多页面命中时先确认，不直接执行高风险操作。

### 5.3 偏航与离线告警

```txt
GPS / 心跳进入 Redis
  → 定时任务读取最新状态
  → RouteDeviationDetector / DeviceOfflineDetector
  → AlertService 去重和落库
  → WebSocket alert.triggered 推送前端
  → 调度员确认/关闭
  → alert_logs 留痕
```

设备期望每 30 秒上报心跳，Redis 心跳 TTL 为 90 秒，检测任务每 30 秒执行一次，因此离线通常在最后心跳后的 90 至 120 秒被发现。单次心跳失败只视为网络抖动，不立即判定离线。

### 5.4 RAG 智能问答

```txt
上传知识文档
  → MinIO 保存原文件
  → 文档切分为 chunks
  → Embedding 写入 pgvector
  → 用户提问
  → 向量检索相关 chunks
  → DeepSeek 生成答案
  → 前端 Markdown 安全渲染
```

## 8. 设计原则

1. **结构化 action 优先**：Agent 不直接执行业务动作，只返回可审计、可确认、可白名单校验的 action。
2. **实时链路与业务链路解耦**：GPS/心跳通过 MQTT + Kafka 入流，业务 API 不阻塞设备上报。
3. **冷热数据分层**：业务数据进 PostgreSQL，时序数据进 TimescaleDB，最新状态进 Redis。
4. **关键操作可追踪**：告警、指令、人脸登录、语音命令都要落日志。
5. **前端只做体验执行层**：页面跳转、地图定位、弹窗由前端负责；权限和数据可信性由后端兜底。
6. **AI 不成为单点依赖**：ASR、TTS、LLM 失败时，位置追踪、人工调度和异常处置仍然可用。
7. **有生命运输优先**：环境异常、宠物不适和交接信息的优先级高于普通运营指标。
