# 智慧萌宠护送平台业务与权限设计

> 版本：v1.0
> 更新时间：2026-07-13
> 对应实现：`pet-safe-transit/smart-logistics-ui/`、`pet-safe-transit/sky-take-out/`
> 文档性质：业务顶层设计、角色权限边界和端到端流程说明

---

## 1. 设计范围

平台在原智慧物流能力上增加萌宠护送业务语义，形成“宠物主人发起和查看、仓管交接与装载、调度员组织运输、司机执行护送、管理员治理系统”的协作闭环。

当前数据库和接口仍沿用 `cargo` 命名，以保持已有代码兼容；产品界面和业务文档使用“萌宠运单”表达。

| 技术实体 | 产品语义 |
|---|---|
| `cargo` | 一次萌宠护送运单及其运输对象 |
| `cargo_vehicle_binding` | 萌宠运单与护送车辆的有效绑定 |
| `cargo_status_logs` | 入库、装载、在途、送达等状态轨迹 |
| `vehicles` | 配备定位和环境采集设备的护送车辆 |
| `device_heartbeats` | 车载定位/环境设备心跳 |
| `alerts` | 离线、偏航、环境异常等风险事件 |
| `commands` | 改道、停车检查、联系货主等调度指令 |

设计边界：系统提供运输可视化、协同调度、风险告警、语音交互和留痕能力，不替代兽医诊断，也不允许 AI 绕过人工确认直接执行高风险调度操作。

## 2. 核心领域对象

| 对象 | 关键内容 | 主要责任角色 |
|---|---|---|
| 用户 | 身份、角色、联系方式、人脸绑定 | 管理员；用户维护本人资料 |
| 萌宠运单 | 宠物信息、起终点、运输状态、关联货主 | 货主、仓管、调度员 |
| 车辆 | 车牌、司机、设备、载重、在线状态 | 仓管、调度员 |
| 车辆绑定 | 运单与车辆的当前关系 | 仓管、调度员 |
| GPS 轨迹 | 经纬度、速度、方向、采集时间 | 设备自动产生，平台只读展示 |
| 环境与心跳 | 温湿度、信号、电量、最后在线时间 | 设备自动产生，仓管/调度员处理异常 |
| 告警 | 类型、级别、状态、位置、处理记录 | 仓管、调度员、管理员 |
| 调度指令 | 指令内容、下发状态、车机回执 | 调度员下发，司机执行 |
| 改址申请 | 新地址、影响评估、审核状态、司机确认 | 货主申请，调度员审核，司机确认 |
| 司机评价 | 分数、标签、评价内容 | 货主提交，调度员查看 |

## 3. 角色职责

### 3.1 货主 `SHIPPER`

- 提供宠物和收发地址信息，发起护送需求。
- 查看本人运单的车辆位置、轨迹、ETA、环境状态和运输时间线。
- 在允许阶段取消运单、申请改址、确认收货并评价司机。
- 使用文字或语音咨询运输进度，但不能查看平台全量告警、车辆调度和人员信息。

### 3.2 仓管 `WAREHOUSE`

- 执行接宠、身份核验、健康/用品交接、入库和装载确认。
- 维护车辆与运单绑定，检查定位和环境设备是否在线。
- 处理仓储、装载和环境类告警并记录处置结果。
- 不负责全局人员管理，也不应自行变更用户角色。

### 3.3 调度员 `DISPATCHER`

- 选择车辆与司机、规划路线、组织发车并监控在途状态。
- 审核影响较大的改址请求，向司机或车机下发调度指令。
- 处理偏航、离线、延误等运输异常，协调仓管、司机和货主。
- 可查看运营所需人员信息，但不能承担系统账号治理职责。

### 3.4 司机 `DRIVER`

- 查看分配给自己的任务、路线和调度指令。
- 执行接宠、发车、途中检查、到达和交付，按节点上报状态。
- 对调度指令回执；出现宠物不适、设备异常或道路风险时主动上报。
- 不能创建车辆、修改他人资料或处理全平台告警。

### 3.5 管理员 `ADMIN`

- 管理用户、角色、系统配置、审计记录和基础数据。
- 具备排障所需的全局可见性，但正常运输应由业务角色执行。
- 管理员权限用于治理和兜底，不代表所有日常操作都应由管理员代办。

## 6. 一次护送的角色协作流程

### 6.1 需求与受理

1. 货主提交宠物信息、起点、终点、联系人和特殊照护要求。
2. 系统生成萌宠运单，初始状态为 `CREATED`。
3. 仓管核对宠物身份、健康说明、随行用品和交接信息。
4. 调度员确认运力、司机和路线，无法满足约束时退回补充信息。

### 6.2 入库、装载与发车

1. 仓管完成接宠和入库检查。
2. 仓管或调度员将运单绑定到车辆，状态进入 `LOADED`。
3. 系统检查车辆、GPS 和环境设备在线情况；异常时阻止直接发车并生成告警。
4. 司机确认任务、宠物和路线后发车，状态进入 `IN_TRANSIT`。

### 6.3 在途监控

1. 车载终端每 30 秒上报心跳，并持续上报 GPS/环境数据。
2. MQTT 接入数据，Kafka 解耦处理，TimescaleDB 保存历史，Redis 保存最新状态。
3. WebSocket 将位置、告警和指令回执实时推送到前端。
4. 货主查看本人宠物位置、ETA 和环境摘要；调度员查看全局运输态势；司机执行任务。

### 6.4 异常处置

1. 系统检测偏航、离线或环境异常，创建告警并执行去重。
2. 仓管处理设备和环境类问题；调度员负责道路、路线和跨角色协调。
3. 调度员下发停车检查、改道或联系货主等指令。
4. 司机执行并回执，处置过程写入 `alert_logs`、`command_logs`。
5. 风险消除后由有权限的角色关闭告警，货主看到经过业务化处理的进展，不暴露内部敏感信息。

### 6.5 到达与结束

1. 司机到达约定地点并完成交接。
2. 货主确认收到宠物，运单进入 `DELIVERED`。
3. 仓管/调度员解除车辆绑定并释放运力。
4. 货主评价司机；系统保留状态、轨迹、告警和调度审计记录。

## 7. 状态机设计

### 7.1 运单状态

```txt
CREATED ──绑定车辆/装载──> LOADED ──司机发车──> IN_TRANSIT ──完成交接──> DELIVERED
   └────────────────────允许阶段取消────────────────────> CANCELLED
```

约束：

- `DELIVERED` 和 `CANCELLED` 是终态，不允许回退到在途状态。
- 解绑后是否回到 `CREATED` 取决于是否已发车；在途运单不得通过普通解绑绕过异常流程。
- 每次状态变化必须写入 `cargo_status_logs`，记录操作人、时间、位置和备注。

### 7.2 告警状态

```txt
PENDING ──确认受理──> ACKNOWLEDGED ──处置完成──> RESOLVED
```

- 同一车辆、同一告警类型在去重窗口内不重复生成大量活动告警。
- 告警关闭必须填写解决方式或备注，不能只改变状态。

### 7.3 调度指令状态

```txt
SENT ──设备收到──> ACKNOWLEDGED ──执行完成──> EXECUTED
  └────────超时/拒绝/设备异常────────> FAILED
```

### 7.4 改址申请状态

```txt
AUTO_APPROVED ──司机确认──> DRIVER_CONFIRMED
PENDING_REVIEW ──调度审核通过──> APPROVED ──司机确认──> DRIVER_CONFIRMED
PENDING_REVIEW ──调度拒绝──> REJECTED
```

## 8. 关键时序

### 8.1 正常位置链路

```txt
车载设备 → EMQX → MQTT Consumer → Kafka → GPS Consumer
         → TimescaleDB（历史） + Redis（最新） → WebSocket → 各角色前端
```

### 8.2 30 秒心跳与离线判定

- 设备期望每 30 秒上报一次心跳。
- Redis 心跳键有效期为 90 秒。
- 离线检测任务每 30 秒扫描一次，因此实际发现时间约为最后心跳后的 90 至 120 秒。
- 单次 30 秒未上报不立即判离线，用三倍心跳周期吸收网络抖动。
- 判定离线后创建去重告警；恢复上报后更新在线状态，并保留历史处置记录。

### 8.3 货主改址

```txt
货主输入新地址 → 高德地理编码 → 计算距离/时延/费用影响
  → 低风险自动通过，或进入调度员审核
  → 生成新路线 → 司机确认 → 更新卸货地址和运输时间线
```

## 9. 语音 Agent 设计

语音入口是交互方式，不是新的权限通道：

```txt
浏览器录音 → /voice/command → 百度 ASR → Agent 意图解析
  → 结构化 Action → 前端执行器 → 路由/地图/结果/确认后的 API
  → 豆包 TTS 生成可爱女声音频回复
```

| Action | 约束 |
|---|---|
| `NAVIGATE` | 必须命中路由白名单，并再次经过路由角色守卫。 |
| `HIGHLIGHT_MAP_TARGET` | 只能定位当前用户有权查看的车辆或运单。 |
| `SHOW_RESULT` | 返回结果必须经过后端数据权限过滤。 |
| `CALL_API` | 仅允许后端白名单接口，前端二次确认，后端再次鉴权。 |
| `NOOP` | 意图不明确时只提示，不猜测执行高风险动作。 |

容错策略：ASR 文本先做车牌、角色页面名和常见同义词归一化；低置信度或多目标命中时要求用户确认，不直接跳转或下发指令。

## 10. 权限实施分层

| 层级 | 当前职责 | 安全作用 |
|---|---|---|
| 菜单过滤 | 按角色只显示可用入口 | 减少误操作，不作为最终鉴权 |
| Vue Router | `meta.roles` 拦截越权页面 | 防止直接输入地址进入页面 |
| Axios/JWT | 自动携带 access token、刷新登录态 | 建立用户身份上下文 |
| 后端拦截器 | 除登录、刷新和公开文件读取外统一校验 JWT | 阻断匿名访问 |
| Controller/Service | `requireAnyRole`、本人或管理员校验 | 阻断角色越权 |
| 数据查询 | 按货主、司机、组织过滤资源 | 目标能力，当前仍需补齐 |
| 审计日志 | 告警、指令、人脸、语音、状态变化留痕 | 支持追责和问题定位 |

## 11. 非功能设计

| 维度 | 设计要求 |
|---|---|
| 可用性 | WebSocket 断线自动重连；实时链路异常时允许 API 轮询最新状态。 |
| 一致性 | PostgreSQL 保存业务事实，TimescaleDB 保存时序历史，Redis 只作最新状态和短期缓存。 |
| 可追踪性 | API 返回 `requestId`；外部 AI/语音调用保留日志 ID；关键操作写审计表。 |
| 性能 | GPS/心跳异步进入 Kafka；地图默认限制轨迹点数量；列表接口分页。 |
| 安全 | 密钥只通过环境变量注入；生产环境启用 HTTPS/WSS、强 JWT 密钥和密码安全哈希。 |
| 隐私 | 宠物主人联系方式、人脸图像和轨迹数据按最小权限展示并设置保留周期。 |
| 降级 | 高德、ASR、TTS、LLM 不可用时返回明确错误，核心追踪和人工调度不依赖 AI 才能工作。 |

## 12. 当前落地与待完善项

### 已落地

- 五类角色、菜单过滤和路由角色守卫。
- 后端 JWT 全局拦截，以及用户、告警、车辆、运单等关键接口的角色校验。
- 萌宠货主首页、运单追踪、车辆调度、司机任务、仓库、告警和智能问答页面。
- MQTT、Kafka、TimescaleDB、Redis、WebSocket 组成的位置与心跳链路。
- 百度 ASR、人脸识别接口、DeepSeek 问答、豆包 TTS 和前端 Action 执行器。
- 改址影响评估、调度审核、司机确认、收货和司机评价接口。

### P0：上线前必须补齐

1. 在运单中增加 `owner_user_id`，在车辆/任务中建立可靠的 `driver_user_id` 关系，后端查询强制行级过滤。
2. 密码从 MD5 升级为 BCrypt 或 Argon2，并轮换已经暴露的 JWT/API 密钥。
3. 将 Agent `CALL_API` 收紧为后端接口、方法、参数三级白名单。
4. 为状态机增加服务端迁移校验，禁止任意状态跳转和终态回退。
5. 为指令、改址审批和告警处理增加幂等键与并发版本控制。

### P1：质量完善

1. 增加角色越权、资源越权、状态机、心跳离线和指令回执自动化测试。
2. 给 Kafka 消费增加重试、死信和消费幂等策略。
3. 完善宠物档案字段，包括品种、年龄、健康提示、温湿度阈值和应急联系人。
4. 增加隐私数据脱敏、审计查询和数据保留/删除策略。

## 13. 验收主链路

1. 货主登录后只能看到萌宠护送、本人追踪和智能问答等授权入口。
2. 仓管完成运单创建/绑定并确认设备在线。
3. 司机发车后，货主页面持续收到位置、ETA 和环境摘要。
4. 设备停止心跳 90 至 120 秒后生成离线告警，仓管/调度员可确认和关闭。
5. 调度员下发改道指令，司机回执后指令状态可追踪。
6. 货主改址经过影响评估、必要审核和司机确认后生效。
7. 到达后货主确认收货并评价，运单、轨迹、告警和操作日志均可追溯。
8. 货主直接访问告警、调度或人员管理接口时，后端返回 403，而不是仅靠前端隐藏。
