# 智慧萌宠护送平台：后端答辩 100 问（详细版）

> 本文严格依据 `pet-safe-transit/sky-take-out` 当前代码整理，重点覆盖后端架构、Spring Boot、MyBatis、PostgreSQL、TimescaleDB、Redis、Kafka、MQTT、WebSocket、权限、Agent、RAG 和第三方 API。回答中会区分“已经实现”和“生产环境仍需完善”的内容。

## 一、后端架构与工程组织（1—10）

### 1. 后端整体采用什么架构？

**参考回答：** 后端是基于 Java 17 运行的 Spring Boot 2.7.3 单体应用，工程采用 Maven 多模块组织，主要分为 `sky-common` 和 `sky-server`。`sky-common` 放通用工具、常量和配置属性，`sky-server` 承载 Controller、Service、Mapper、实时消息、AI 和第三方服务接入。当前没有拆成微服务，原因是答辩项目业务规模有限，单体架构更容易保证事务、部署和联调稳定；但业务包已经按领域分层，后续可拆分设备接入、轨迹、告警和 AI 服务。

### 2. 为什么项目暂时没有采用微服务？

**参考回答：** 当前团队和部署资源有限，若拆成多个服务，会额外引入注册中心、配置中心、链路追踪、服务间鉴权和分布式事务，复杂度会明显高于收益。项目真正需要独立扩展的是 MQTT/Kafka 实时链路和 AI 调用，但现阶段通过模块边界与异步消息已经实现了解耦。答辩时应说明这是基于规模的架构选择，不是认为微服务一定更先进。

### 3. `sky-common` 和 `sky-server` 分别负责什么？

**参考回答：** `sky-common` 是可复用基础模块，包含 JWT、HTTP、文件、对象存储等工具以及配置属性类；`sky-server` 依赖 `sky-common`，提供实际业务服务。这样可以避免 Controller、Service 重复实现基础能力，也为以后拆服务保留公共依赖。但要注意公共模块不能无限膨胀，业务实体和业务规则仍应留在 `sky-server` 的物流领域包中。

### 4. 后端为什么按 Controller、Service、Mapper 分层？

**参考回答：** Controller 负责 HTTP 协议、参数接收和响应封装；Service 负责业务规则、状态流转和事务；Mapper 负责 SQL 与数据库映射。分层后，接口变化不会直接污染 SQL，数据库变化也不必重写 Controller。项目中的 `CargoController -> CargoServiceImpl -> LogisticsCargoMapper.xml` 就是完整示例。

### 5. DTO、Entity、VO 的区别是什么？

**参考回答：** DTO 表示接口输入，例如 `CargoCreateDTO`、`RoutePlanRequestDTO`；Entity 对应数据库记录，例如 `Vehicle`、`Alert`；VO 表示输出给前端的数据，例如 `VehicleVO`、`CargoVO`。三者分开可以防止前端直接控制数据库字段，也避免把密码哈希、内部主键或审计字段无意返回。项目之前出现过 `Map<String,Object>` 较多的问题，新代码已在关键业务逐步使用 DTO/VO，但部分统计和动态 Agent 结果仍保留 Map。

### 6. 为什么部分接口还使用 `Map<String,Object>`？

**参考回答：** 告警统计、Agent Action、外部 API 返回和动态时间线的字段结构变化较大，Map 能降低初期联调成本。例如 Agent 的不同动作可能携带车牌、订单号、路由或确认文案，难以用一个固定对象表达。不过 Map 会削弱编译期类型检查，因此稳定接口应继续改成明确 DTO/VO，动态扩展字段才保留 Map。

### 7. 项目主要后端依赖有哪些？

**参考回答：** 核心依赖包括 Spring Web、原生 MyBatis Starter 2.2.0、PostgreSQL JDBC 42.3.6、Druid、Spring Data Redis、Spring Kafka、Spring Integration MQTT、Spring WebSocket、MinIO SDK、pgvector JDBC、Knife4j、Lombok 和 JJWT。项目没有使用 MyBatis-Plus，SQL 主要由 Mapper 接口和 XML 编写。

### 8. 为什么使用构造器注入？

**参考回答：** 多数新业务类通过构造器注入依赖，例如 `CargoServiceImpl` 注入两个 Mapper。构造器注入可以让依赖不可变，实例创建时就保证依赖完整，也更方便单元测试传入 Mock。它比字段上的 `@Autowired` 更容易看清类的真实依赖关系。

### 9. 后端如何启动和打包？

**参考回答：** 在 `sky-take-out` 下执行 `mvn -pl sky-server -am -DskipTests package`，Maven 会先构建父工程和 `sky-common`，再由 Spring Boot Maven Plugin 将 `sky-server` 重打包为可执行 JAR。运行命令是 `java -jar sky-server/target/sky-server-1.0-SNAPSHOT.jar`，默认监听 8080。当前本机业务 PostgreSQL 实际使用 5432，启动时需要用 `POSTGRES_PORT=5432` 覆盖源码默认值 15432。

### 10. 当前架构最值得继续重构的地方是什么？

**参考回答：** 第一是将旧外卖项目遗留包与智慧物流领域彻底分离；第二是把大量动态 Map 收敛成 DTO；第三是统一权限注解或策略组件，而不是各 Controller 手写角色数组；第四是为设备接入、告警和 AI 增加自动化测试。若业务量增长，再把 MQTT 接入、轨迹查询和 AI 服务拆成独立进程。

## 二、Spring Boot 与接口设计（11—20）

### 11. 为什么选择 Spring Boot 2.7.3？

**参考回答：** Spring Boot 能自动装配 Web、Redis、Kafka、配置绑定和数据源，减少基础工程代码。2.7.3 与项目现有 Java 17、Springfox/Knife4j 3.x 和依赖版本兼容。它不是最新版本，生产升级到 Spring Boot 3 时需要把 `javax.*` 迁移到 `jakarta.*`，并重新验证 Knife4j、JJWT 和 MQTT 依赖。

### 12. API 为什么统一使用 `/api/v1` 前缀？

**参考回答：** `/api/v1` 表示第一版稳定接口，前端通过 Vite 或 Nginx 将 `/api` 代理到后端。版本前缀允许未来新增 `/api/v2`，而不立即破坏旧前端。部分订单和路线 Controller 还保留 `/api/...` 兼容路径，这是迁移期设计，最终应统一为一个版本规范。

### 13. `ApiResponse` 统一响应包含哪些内容？

**参考回答：** 响应包含业务 `code`、`message`、`data`、`timestamp` 和 `requestId`。成功时通常 `code=0`，失败时由异常处理器返回认证、权限、参数或服务器错误码。`requestId` 能让前端报错与后端日志关联，比只返回 HTTP 状态更利于排查。

### 14. HTTP 状态码和业务状态码为什么要同时存在？

**参考回答：** HTTP 状态码描述协议层结果，例如 200、403、500；业务码进一步区分“Token 失效”“权限不足”“参数错误”等具体情况。前端可以先按 HTTP 状态处理通用逻辑，再按业务码显示业务提示。当前项目权限异常会返回 HTTP 403 和业务码 `40301`。

### 15. 全局异常是怎样处理的？

**参考回答：** `LogisticsExceptionHandler` 使用 `@RestControllerAdvice` 捕获认证异常、权限异常、参数异常和未预期异常，并转换成统一 `ApiResponse`。这样 Controller 不需要重复写 try/catch。未预期异常只应在日志记录堆栈，对客户端返回受控信息，避免泄露 SQL、路径和密钥。

### 16. 分页接口怎样限制请求规模？

**参考回答：** 用户、车辆、货物、告警和知识文档服务都设置默认页码 1、默认每页 20 条、最大每页 100 条。Service 会标准化非法页码和 size，再计算 offset。限制最大 size 可以防止用户传入几十万条造成数据库、网络和 JVM 内存压力。

### 17. Knife4j 在项目中有什么作用？

**参考回答：** `WebMvcConfiguration` 创建 Docket，只扫描 `com.sky.logistics.controller` 下的 `/api/v1/**`，生成可交互接口文档，入口是 `/doc.html`。配置还声明全局 Authorization Header，方便登录后测试 JWT 接口。接口文档用于联调，不应在公网生产环境无保护开放。

### 18. 项目如何处理跨域？

**参考回答：** 当前 `WebMvcConfiguration` 对 `/api/v1/**` 允许常用 HTTP 方法、任意 Header 和任意 Origin Pattern，并允许携带凭证。本地开发方便，但生产环境过宽，应改成明确域名白名单。更推荐前端和 API 通过同一 Nginx 域名发布，从根源上减少 CORS 配置。

### 19. 为什么扩展 Jackson 消息转换器？

**参考回答：** 项目把自定义 `JacksonObjectMapper` 放到转换器列表首位，用于统一 Java 对象与 JSON 的转换规则，例如日期时间格式和空值处理。统一转换器可以避免每个 Controller 手工格式化时间。需要注意自定义 Mapper 要与 Spring Boot 默认模块兼容，特别是 Java Time 类型。

### 20. Controller 为什么不直接操作数据库？

**参考回答：** Controller 直接调用 Mapper 会把协议逻辑、业务规则和持久化耦合在一起，也难以设置完整事务。比如绑定宠物运输记录与车辆，需要校验订单状态、校验车辆存在、写绑定表并返回组合数据，这些应由 `CargoServiceImpl` 完成。Controller 只负责角色检查和参数转发。

## 三、MyBatis、PostgreSQL 与事务（21—35）

### 21. 项目使用 MyBatis 还是 MyBatis-Plus？

**参考回答：** 项目实际使用原生 `mybatis-spring-boot-starter:2.2.0`，没有 MyBatis-Plus。证据是 Mapper 使用 `@Mapper`，SQL 位于 `resources/mapper/*.xml`，代码中不存在 `BaseMapper`、`IService` 或 MyBatis-Plus Wrapper。答辩时不要把二者说混。

### 22. 为什么选择原生 MyBatis？

**参考回答：** 项目存在 PostgreSQL 特有 SQL、pgvector 相似度检索、复杂关联、状态统计和 TimescaleDB 查询，需要精确控制 SQL。原生 MyBatis XML 能清楚展示字段、索引条件和动态查询，便于解释与调优。MyBatis-Plus 对标准单表 CRUD 更方便，但不会消除这些复杂 SQL。

### 23. Mapper 接口和 XML 如何关联？

**参考回答：** XML 的 `<mapper namespace>` 写 Mapper 接口全限定名，语句 `id` 对应接口方法名。`application.yml` 将 mapper-locations 配置为 `classpath:mapper/*.xml`，并开启下划线到驼峰映射。启动时 MyBatis 创建接口代理，调用接口方法时执行对应 SQL。

### 24. 为什么业务库使用 PostgreSQL？

**参考回答：** PostgreSQL 支持事务、约束、复杂 SQL、JSON 和扩展机制，同时 TimescaleDB、pgvector 都建立在 PostgreSQL 生态上。这样业务数据、时序数据和向量检索使用相近 SQL 与运维工具。当前主业务库是 `smart_logistics`，保存用户、车辆、运输记录、告警、指令和知识库元数据。

### 25. 主业务库有哪些核心表？

**参考回答：** 核心表包括 `users`、`vehicles`、`cargo`、`cargo_vehicle_binding`、`cargo_status_logs`、`alerts`、`alert_logs`、`commands`、`command_logs`、`knowledge_documents`、`knowledge_chunks`、`user_face_bindings`、`face_login_logs` 和 `voice_command_logs`。订单扩展部分还有改址申请、司机评价和卸货地址记录表。

### 26. 为什么运输记录和车辆使用中间绑定表？

**参考回答：** `cargo_vehicle_binding` 保存某个运输记录在某段时间绑定哪辆车，而不是只在 cargo 表保存车牌。这样可以保留换车和历史关系，也支持一辆车同时承载多个宠物运输订单。当前通过 `ACTIVE` 状态和部分唯一索引约束有效绑定，解绑不会删除历史记录。

### 27. 为什么还要保存 `cargo_status_logs`？

**参考回答：** cargo 表只保存当前状态，状态日志表记录每次状态变化的时间、位置、备注和操作人。它支持运输时间线、审计和异常追溯。例如从 CREATED 到 LOADED、IN_TRANSIT、DELIVERED 的全过程不能仅靠当前状态还原。

### 28. 项目怎样保证车牌和 IMEI 唯一？

**参考回答：** `VehicleServiceImpl` 创建和更新车辆前，会通过 Mapper 检查车牌及设备 IMEI 是否已存在，更新时使用排除当前 ID 的查询。数据库层还应保留唯一约束，避免两个并发请求同时通过应用校验。应用校验提供友好提示，数据库约束才是最终一致性保障。

### 29. 车辆为什么不能随意删除？

**参考回答：** 删除前 `VehicleServiceImpl` 会检查车辆是否存在，并检查是否仍有有效运输绑定。如果有绑定就拒绝删除，防止订单失去承运关系。生产环境更适合软删除或停用状态，因为车辆和历史轨迹具有审计价值。

### 30. `@Transactional(readOnly=true)` 有什么作用？

**参考回答：** `CargoServiceImpl`、`VehicleServiceImpl` 和 `UserServiceImpl` 类级别使用只读事务，查询方法默认不允许写入意图；具体写方法再覆盖为普通 `@Transactional`。这能表达方法语义，并可能帮助连接池或数据库优化。它不是绝对安全边界，真正的写保护还依赖数据库账号权限和代码审查。

### 31. 哪些操作必须放在同一事务里？

**参考回答：** 运输状态更新需要同时更新 cargo 当前状态和写状态日志；绑定需要写绑定表并更新相关业务状态；创建告警需要写告警主表和告警日志；知识文档索引需要批量写 chunk 并更新文档状态。这些步骤中任意一步失败都应回滚，否则会出现“主状态已变但没有审计记录”的不一致。

### 32. 当前双数据源事务有什么限制？

**参考回答：** 主业务库由 Druid 和 `DataSourceTransactionManager` 管理，是 `@Primary` 数据源；TimescaleDB 使用独立 HikariDataSource 和 JdbcTemplate。普通 `@Transactional` 只覆盖主业务库，不会自动把 TimescaleDB 写入纳入同一事务。当前通过 Kafka 异步写时序库接受最终一致性；若要求跨库强一致，需要分布式事务、Outbox 或补偿机制。

### 33. 如何防止 SQL 注入？

**参考回答：** Mapper XML 中用户输入应使用 `#{}` 参数绑定，而不是 `${}` 字符串拼接。动态排序列不能直接接受前端文本，应先映射到后端白名单。Agent 参数同样是不可信输入，必须经过 DTO 校验和业务白名单后再进入 Mapper。

### 34. 数据库索引是怎样设计的？

**参考回答：** 告警按 status、vehicle_plate、triggered_at 建索引；指令按 plate 和 created_at 建索引；状态日志按 cargo_id 和时间建索引；人脸绑定按 user_id 与百度 group/user 建索引。索引围绕高频过滤条件和时间倒序查询设计，但不能无限添加，因为每个索引都会增加写入成本。

### 35. 当前密码存储有什么问题？

**参考回答：** 当前 `AuthServiceImpl` 和 `UserServiceImpl` 使用 MD5 处理密码，这是演示项目的明显安全不足。MD5 速度太快且没有自适应成本，不能抵抗彩虹表和暴力破解。生产环境应迁移到 BCrypt、SCrypt 或 Argon2，并为旧账号设计登录后渐进升级方案。

## 四、双数据库、缓存与实时消息（36—50）

### 36. 为什么同时使用 PostgreSQL 和 TimescaleDB？

**参考回答：** PostgreSQL 主库保存强业务关系和状态，TimescaleDB 保存大量按时间追加的 GPS 点和设备心跳。轨迹数据主要按车辆和时间范围查询，TimescaleDB 的 hypertable 更适合时间分区和时序索引。拆开后，轨迹高频写入不会直接影响用户、订单和告警事务。

### 37. TimescaleDB 中有哪些表？

**参考回答：** 当前主要有 `gps_points` 和 `device_heartbeats` 两张超表。`gps_points` 按时间和 vehicle_id 查询，保存经纬度、速度、方向、精度等；`device_heartbeats` 按时间和 IMEI 查询，保存电量、信号、卫星数和温度等。SQL 使用 `create_hypertable` 将普通表转换为时序超表。

### 38. 两个数据源怎样区分？

**参考回答：** `DataSourceConfig` 创建名为 `dataSource` 的主数据源并标记 `@Primary`，MyBatis 默认使用它；`TimescaleDBConfig` 创建 `timescaleDataSource` 和 `timescaleJdbcTemplate`。需要访问轨迹库的 Service 或 Kafka Consumer 通过 `@Qualifier("timescaleJdbcTemplate")` 精确注入，避免误写主库。

### 39. 为什么主库用 Druid，时序库用 Hikari？

**参考回答：** 主库沿用项目已有 Druid 配置并接入 MyBatis 事务；时序库是后增的独立数据源，使用轻量 Hikari 手工配置最大连接 5、最小空闲 1、连接超时 10 秒。两种连接池并存不是必须设计，后续可以统一以降低运维认知成本，但必须继续保留 Bean 名和事务边界区分。

### 40. Redis 在项目中存了什么？

**参考回答：** Redis 保存车辆最新位置、设备在线心跳、告警去重键、偏航检测去重键以及智能问答会话。最新位置键为 `logistics:vehicle:latest:{plate}`，有效期 24 小时；设备心跳键为 `logistics:device:heartbeat:{imei}`，有效期 90 秒；问答会话有效期 1 小时。Redis 用于热数据和临时状态，历史事实仍写数据库。

### 41. 为什么最新位置放 Redis？

**参考回答：** 地图和 ETA 会频繁查询最新点，如果每次扫描时序库会增加延迟和数据库压力。GPS Consumer 在写 TimescaleDB 后同步刷新 Redis，查询优先读 Redis，缺失时再回退到 TimescaleDB 最新记录。这个模式兼顾实时速度和历史可靠性。

### 42. MQTT 在系统中的职责是什么？

**参考回答：** MQTT 负责车载终端与平台之间的轻量实时通信。后端订阅 `vehicle/+/gps` 和 `vehicle/+/heartbeat`，QoS 为 1；下发调度指令时通过出站 Channel 发布到车辆命令主题。`+` 是单层通配符，用于匹配不同车辆的 vinTopic。

### 43. MQTT QoS 1 意味着什么？

**参考回答：** QoS 1 保证消息至少到达一次，因此可能出现重复消息。系统不能把“收到一次”当作天然幂等，GPS 可以按时间和设备标识去重，命令回执要按 commandId 更新。它在可靠性和成本之间比 QoS 0 更适合运输监控，但没有 QoS 2 那么重。

### 44. Kafka 在实时链路中解决什么问题？

**参考回答：** MQTT Receiver 不直接完成所有数据库写入，而是把 GPS 发送到 `gps-points`，把心跳发送到 `vehicle-heartbeats`。Kafka 把设备接入与持久化解耦，流量突增时可以缓冲，消费者失败后可重新消费。当前两个 Consumer 使用同一消费组并分别监听对应 Topic。

### 45. 一条 GPS 数据的完整链路是什么？

**参考回答：** 车载设备发布 MQTT GPS 消息；`MqttMessageReceiver` 提取 vinTopic、反序列化并校验经纬度；消息发送到 Kafka `gps-points`；`GpsKafkaConsumer` 根据 vinTopic 找到车辆，写入 TimescaleDB；随后刷新 Redis 最新位置，并通过 WebSocket 广播 `vehicle.position`。前端收到事件后更新地图 Marker。

### 46. 一条心跳数据如何处理？

**参考回答：** MQTT Receiver 收到 `/heartbeat` 后发送到 Kafka `vehicle-heartbeats`，同时立即写 Redis 在线键并设置 90 秒 TTL。`HeartbeatKafkaConsumer` 将电量、信号、卫星数和温度写入 TimescaleDB。Redis 负责快速在线判断，TimescaleDB 负责心跳历史查询。

### 47. 30 秒心跳失败后会立即离线吗？

**参考回答：** 不会。30 秒是期望上报和检测周期，Redis 在线键 TTL 为 90 秒，所以允许约三次心跳窗口，避免短暂网络抖动误报。`DeviceOfflineDetector` 每 30 秒扫描一次，键过期后将设备状态改为 OFFLINE 并生成严重告警，因此实际判离线约在 90 至 120 秒。

### 48. 设备恢复心跳后怎样处理？

**参考回答：** 新心跳会重新创建 Redis 在线键，下一次离线检测发现键存在后，把数据库设备状态恢复为 ONLINE。当前代码会记录恢复在线日志，但历史离线告警不会自动当作已解决，仍需要业务人员确认或增加自动恢复规则。这样能避免恢复瞬间把需要复盘的告警直接抹掉。

### 49. WebSocket 推送了哪些事件？

**参考回答：** 当前主要广播 `vehicle.position`、`alert.triggered` 和 `command.ack`。WebSocket Server 维护会话与订阅关系，前端订阅后实时更新车辆位置、告警列表和指令状态。断线时前端会指数退避重连，并用 HTTP 轮询作为降级。

### 50. 实时链路如何处理消息重复和乱序？

**参考回答：** 当前已有业务 ID、时间戳、Kafka offset 和告警去重键等基础条件，但完整的事件幂等表与乱序窗口还未系统实现。生产方案应给每条设备事件增加 eventId 和 deviceTimestamp，数据库建立唯一约束，旧时间戳不能覆盖新状态；对命令回执按 commandId 做状态机校验。

## 五、运输业务、路线与告警（51—65）

### 51. 运输记录有哪些状态？

**参考回答：** 当前 `CargoServiceImpl` 允许 `CREATED`、`LOADED`、`IN_TRANSIT`、`DELIVERED` 和 `CANCELLED`。创建时默认 CREATED，每次更新同时写状态日志。当前只校验状态值是否合法，生产环境还应建立显式状态转换表，例如 DELIVERED 不能回到 IN_TRANSIT。

### 52. 为什么已送达或已取消记录不能再修改？

**参考回答：** DELIVERED 和 CANCELLED 属于终态，继续修改路线、重量或绑定车辆会破坏审计事实。`CargoServiceImpl` 在更新和绑定前检查终态并拒绝操作。若确需纠错，应通过更正记录或管理员审计流程，而不是直接覆盖历史。

### 53. 取消运输记录时做了什么？

**参考回答：** Service 先校验记录存在且未送达，然后把状态改为 CANCELLED，写状态日志，并将当前 ACTIVE 车辆绑定改为解绑状态。整个过程放在事务中，避免订单取消但车辆仍显示承运。接口角色允许货主、调度员和管理员调用，但生产还需校验货主只能取消自己的订单。

### 54. 如何查询宠物当前所在位置？

**参考回答：** `TrackingServiceImpl` 先根据 cargoId 查有效车辆绑定，再用车牌查询 Redis 最新位置。如果 Redis 没有数据，则查询 TimescaleDB 最新 GPS 点；没有绑定返回 NO_BINDING，没有轨迹返回 NO_GPS。这样接口能明确区分业务未调度和设备暂无数据。

### 55. 历史轨迹为什么限制最多 200 点？

**参考回答：** 地图一次返回过多坐标会增加 SQL、网络传输和前端绘制成本，所以 `TRAJECTORY_MAX_POINTS=200`。这是简单限流策略，大轨迹应按时间范围查询并做抽稀，例如 Douglas-Peucker 算法或 TimescaleDB 时间桶聚合。

### 56. ETA 是怎样计算的？

**参考回答：** 当前先用 Haversine 公式计算当前位置到目的地的球面距离，再根据当前速度或默认速度估算时间，并乘道路修正系数。返回剩余距离和预计到达时间。它是可解释的基础算法，但未考虑实时路况、休息、天气和车辆限行，因此项目还接入高德路线服务用于更准确规划。

### 57. 为什么 Haversine 公式不能直接代表道路距离？

**参考回答：** Haversine 只计算球面两点最短距离，相当于直线距离，不知道公路形状、桥梁、限行和拥堵。代码通过 roadFactor 做粗略修正，但精度有限。生产 ETA 应优先使用高德驾驶路线距离和时长，并在外部 API 失败时回退到 Haversine。

### 58. 高德地图后端封装了哪些能力？

**参考回答：** `AmapServiceImpl` 提供输入提示、地理编码、逆地理编码和路线规划，并配置连接与读取超时。路线服务支持普通规划、货车规划、重新规划和偏航检查。外部调用统一封装，Controller 不直接拼高德 URL，便于缓存、降级和替换供应商。

### 59. 系统如何判断偏航？

**参考回答：** `RoutePlanServiceImpl` 计算当前位置到规划折线各线段的最小距离，默认阈值为 3 公里。超过阈值判定偏航，超过两倍阈值标为 HIGH，否则为 MEDIUM；阈值内为 LOW。这个规则简单透明，后续可以结合道路等级、定位精度和连续多点判断减少漂移误报。

### 60. 自动偏航检测多久执行一次？

**参考回答：** `RouteDeviationDetector` 初始延迟 30 秒，之后每 60 秒执行一次，扫描运输中的车辆。发现偏航后会创建告警，并设置 1 小时去重键，避免每分钟重复通知。定时任务适合当前规模，车辆数很大时应改为基于 GPS 流的事件计算。

### 61. 告警为什么要做去重？

**参考回答：** 设备持续离线或车辆持续偏航时，检测器会周期性命中同一异常。如果每次都插入告警，页面会被重复记录淹没。`AlertServiceImpl` 使用 `车辆+告警类型` 作为 Redis 去重键，TTL 24 小时；告警关闭时主动删除该键，使后续再次发生时能够重新告警。

### 62. 告警状态怎样流转？

**参考回答：** 新告警状态是 PENDING；处理人员确认后变为 ACKNOWLEDGED；给出处理结果后变为 RESOLVED。每次状态变化写 `alert_logs`，保存操作和备注。Service 会阻止确认非 PENDING 告警，也会阻止重复关闭已解决告警。

### 63. 调度指令如何下发？

**参考回答：** 调度端调用车辆 command 接口，`CommandServiceImpl` 先在 `commands` 和 `command_logs` 中保存指令及 SENT 状态，再通过 MQTT Outbound Channel 发布消息。设备执行后发送 `/command/ack`，后端更新状态并通过 WebSocket 通知前端。先落库再发送能保留审计，但还应为发送失败增加重试和失败状态。

### 64. 改址申请为什么要先计算影响？

**参考回答：** `OrderExtensionServiceImpl` 分别规划旧路线和新路线，计算额外距离、预计延误、额外费用以及新地址偏离原路线的程度。额外超过 50 公里或离原路线超过 30 公里判定超服务区；中高影响需要调度审核。这样把“能不能改”变成可解释规则，而不是直接覆盖目的地。

### 65. 系统怎样验证“已送达”状态可信？

**参考回答：** 状态核验接口计算上报位置到目的地的 Haversine 距离，如果声称 DELIVERED 但距离大于 1.5 公里，则判定可信度不足。这个规则能拦截明显错误，但定位漂移、园区入口和人工签收仍可能造成误差，生产应结合电子围栏、停留时长和签收凭证。

## 六、认证、权限与安全（66—76）

### 66. 登录流程是什么？

**参考回答：** `/auth/login` 根据用户名查询 users 表，校验密码后生成 accessToken 和 refreshToken，并返回用户角色与权限标识。前端后续在 Authorization Header 中携带 Bearer Token。访问 `/users/me` 可重新获得当前用户资料，避免只信任浏览器本地角色。

### 67. JWT 中保存了哪些信息？

**参考回答：** 当前 Token Claims 包含 userId、username、name、role 和过期时间。后端从 Token 恢复用户上下文，用于角色判断和审计记录。JWT 中不应放密码、API Key 或频繁变化的大对象，因为 Token 会在客户端保存且签发后不能直接修改。

### 68. accessToken 和 refreshToken 有什么区别？

**参考回答：** accessToken 生命周期较短，用于每次 API 请求；refreshToken 生命周期较长，只用于换取新 Token。这样 accessToken 泄露后的有效窗口较小，用户也不必频繁登录。当前项目已经有刷新接口，但生产还需要服务端撤销列表、设备会话管理和 refreshToken 轮换。

### 69. 哪些接口不需要登录？

**参考回答：** `LogisticsAuthInterceptor` 当前放行账号登录、人脸登录、刷新 Token、OPTIONS 预检请求，以及公开图片 GET。其他 `/api/v1/**` 和兼容 `/api/**` 都要求有效 JWT。文件上传仍需登录，只有已生成图片的读取路径公开，便于 `<img>` 直接显示。

### 70. 为什么前端隐藏菜单不等于权限控制？

**参考回答：** 用户可以手工构造 HTTP 请求，绕过菜单和路由守卫。真正权限由后端拦截器确认身份，再由 Controller 或 Security Service 检查角色。例如告警接口只允许 WAREHOUSE、DISPATCHER、ADMIN，即使 DRIVER 知道 URL 也会收到 403。

### 71. 当前有哪些角色？

**参考回答：** 系统定义 SHIPPER、WAREHOUSE、DISPATCHER、DRIVER 和 ADMIN。货主主要查询订单和位置，仓管维护运输与绑定并处理告警，调度员管理车辆和指令，司机查看任务并上报状态，管理员管理用户与全局资源。角色来自后端 JWT，不允许真实 API 模式在前端任意切换。

### 72. 如何防止普通用户把自己改成管理员？

**参考回答：** 用户更新接口先通过 JWT 获得当前用户，只允许本人或管理员修改目标账号。非管理员请求若携带不同 role，后端明确抛出 403，而不是只依赖前端禁用下拉框。用户列表、创建、删除和角色分配都只允许 ADMIN。

### 73. 当前权限模型还有什么不足？

**参考回答：** 已实现角色级接口控制，但细粒度权限标识主要用于展示，尚未统一变成注解或策略执行；更重要的是数据范围隔离还不完整，例如货主查询列表时应只返回自己的订单。生产应在表中增加 ownerId/customerId，并在 Service 或数据权限拦截器中强制过滤。

### 74. 人脸登录的处理流程是什么？

**参考回答：** 前端上传 Base64 正脸图；后端调用百度人脸搜索，在配置 group 中寻找候选人；相似度达到阈值 80 才通过；再根据百度 user_id 找本地绑定用户，签发平台 JWT，并写人脸登录日志。本地保存的是绑定关系与标准照路径，不应把百度搜索结果直接当作系统账号。

### 75. 人脸标准图片存在哪里？

**参考回答：** `FaceImageStorageService` 将标准照保存到本地上传目录，并在 `user_face_bindings` 保存路径、百度 groupId 和 baiduUserId。单张图片最大限制 3 MB。生产环境更适合私有对象存储、服务端加密和临时签名 URL，并设置删除与留存策略。

### 76. 文件上传有哪些安全要求？

**参考回答：** 当前知识文档限制 10 MB，人脸图片限制 3 MB，普通图片接口也检查类型并生成服务端文件名。生产还应验证文件魔数而不只信 Content-Type，阻止路径穿越，隔离上传目录，执行病毒扫描，并限制每个账号的频率和总容量。

## 七、AI、RAG、语音与 Agent（77—90）

### 77. 智能问答后端链路是什么？

**参考回答：** `AssistantController` 接收问题和 sessionId，`AssistantServiceImpl` 识别当前角色、抽取订单或车牌上下文、检索知识片段，构造系统 Prompt，再调用 `LLMService`。结果返回 answer、sessionId 和 sources，并把问答写入 Redis 会话列表，TTL 为 1 小时。

### 78. 项目使用什么大模型？

**参考回答：** `application-dev.yml` 当前配置 DeepSeek OpenAI 兼容接口，模型是 `deepseek-chat`。`LLMService` 封装普通对话、流式对话和工具选择，因此业务 Service 不直接依赖供应商 HTTP 细节。API Key 当前仍有硬编码位置，生产应全部迁移到环境变量和密钥管理服务。

### 79. RAG 为什么需要知识文档表和切片表？

**参考回答：** `knowledge_documents` 保存文档标题、分类、对象存储 Key、状态和切片数；`knowledge_chunks` 保存每个文本片段及向量。文档级元数据负责管理生命周期，切片级数据负责检索。分表后可以重新索引单个文档，也能返回来源信息。

### 80. 知识文档怎样建立索引？

**参考回答：** 文件先上传到 MinIO并登记文档；索引时读取文本，按段落和最大 800 字符切片，为每个切片请求 embedding，然后写入 `knowledge_chunks`，最后把文档状态改为 INDEXED。批量写 chunk 与状态更新放在事务方法中，失败时标记错误而不是假装索引成功。

### 81. pgvector 在项目中怎样使用？

**参考回答：** `knowledge_chunks.embedding` 是向量列，建立 `ivfflat` 余弦索引，Mapper 通过向量距离查询相似片段。pgvector 让向量检索和业务元数据留在 PostgreSQL 中，部署比独立向量库简单。数据规模变大后需要重新评估 lists、探测参数、召回率和索引重建策略。

### 82. 什么是混合检索？

**参考回答：** 普通问答同时执行 embedding 向量检索和关键词检索，再使用 RRF 融合排名。项目 `TOP_K=5`、`RRF_K=60`，分别按排名计算分数并合并相同 chunk。向量擅长语义相似，关键词擅长车牌、订单号等精确文本，两者结合比单一路径稳定。

### 83. 为什么流式问答暂时只做关键词检索？

**参考回答：** `chatStream` 为了让首个模型 Token 更快显示，当前跳过可能耗时的 embedding 请求，先做关键词检索再流式调用 LLM。这是延迟与召回率的权衡。后续可以并行执行向量检索、缓存常见 embedding，或者在 ready 事件后异步补充来源。

### 84. 问答如何降低幻觉？

**参考回答：** 系统 Prompt 注入当前角色、业务上下文和检索到的资料，并要求模型基于资料回答；响应同时返回 chunk 来源供前端展示。涉及真实操作时不让模型直接执行 SQL，而是输出白名单 Action。仍需增加“无证据就明确不知道”、来源阈值和答案事实校验。

### 85. 百度语音识别如何接入？

**参考回答：** `BaiduAsrService` 接收 MultipartFile，将音频转成 Base64，携带格式、采样率、长度、设备标识和 OAuth Token 请求百度 ASR。连接超时 8 秒、读取超时 90 秒，识别成功后取第一条文本。浏览器录音格式必须与请求声明一致，否则会出现识别错误或空结果。

### 86. 为什么语音识别需要容错？

**参考回答：** 车牌、订单号和物流术语容易被同音字误识别。后端通过正则提取、文本归一化、规则意图优先和 LLM Tool Calling 兜底提高容错；高风险操作还要求二次确认。进一步可增加车牌候选模糊匹配、拼音相似度和基于当前用户订单的候选集合。

### 87. 豆包 TTS 怎样生成语音？

**参考回答：** `DoubaoTtsService` 向 Seed Audio HTTP 接口发送 model、text_prompt、speaker reference 和 audio_config，接口返回 Base64 音频后解码为 MP3 字节。读取超时设置为 120 秒以覆盖最长生成时间。当前使用可爱女声音色 `zh_female_vv_uranus_bigtts`，但账号必须在火山控制台获得对应资源授权。

### 88. TTS 失败时系统怎么办？

**参考回答：** 前端优先请求后端豆包音频；接口失败时仍显示文字答案，并可降级到浏览器 `speechSynthesis` 女声。第三方语音失败不应导致 Agent 文本结果丢失。后端还应记录供应商 logId、错误码、耗时，并对可重试错误做有限次数重试。

### 89. Agent 如何把自然语言转成操作？

**参考回答：** `VoiceAgentServiceImpl` 先用正则、关键词和角色规则识别常见意图；无法明确匹配时，调用 `LLMService.chooseTool`，让模型从 `AgentToolCatalog` 声明的函数中选择。结果转换成 `AgentActionVO` 返回前端，前端执行页面跳转、地图定位、弹窗或 API 调用。

### 90. 为什么 Agent 操作必须有工具白名单和确认？

**参考回答：** 大模型输出不稳定，不能允许它任意生成 URL 或 SQL。`AgentToolCatalog` 明确每个工具的名称、说明、参数类型和必填字段；只有查询和导航属于只读集合，其余默认要求确认。最终后端接口仍执行 JWT 和角色校验，模型的工具选择不是授权凭证。

## 八、可靠性、测试、部署与改进（91—100）

### 91. 外部服务如何设置超时？

**参考回答：** 高德连接默认 3 秒、读取 5 秒；Embedding 连接 5 秒、读取 30 秒；LLM 连接 5 秒、读取 60 秒；百度 ASR 读取 90 秒；豆包非流式 TTS 读取 120 秒。超时要根据业务耗时设置，不能无限等待，也不能所有服务统一一个值。下一步应增加重试、熔断和指标监控。

### 92. 如果 Redis 故障会发生什么？

**参考回答：** 最新位置查询会回退 TimescaleDB，但设备在线判断、告警去重、会话和偏航去重会受影响。当前部分代码会捕获 Redis 异常，但还没有统一降级策略。生产可配置 Redis Sentinel/Cluster，并把关键幂等信息在数据库保留兜底。

### 93. 如果 Kafka 故障会发生什么？

**参考回答：** MQTT Receiver 仍可能收到设备消息，但发送 Kafka 失败后，GPS 与心跳历史不能可靠写入 TimescaleDB，WebSocket实时更新也可能中断。应检查 send Future、记录失败消息并进入本地重试或死信队列。生产还需监控 Topic lag、消费失败率和分区状态。

### 94. 如果 MQTT Broker 断开会怎样？

**参考回答：** Paho 配置了 automaticReconnect 和 60 秒 KeepAlive，会尝试自动重连；断开期间设备消息无法进入后端，调度指令也无法下发。恢复后是否补数据取决于设备端会话、QoS 和离线缓存配置。关键设备应在终端保存待上报事件并携带原始时间戳补传。

### 95. 当前日志和可观测性够不够？

**参考回答：** 代码已记录服务启动、外部请求、MQTT/Kafka处理、设备恢复、告警创建和错误堆栈，API 响应也带 requestId。但缺少统一结构化日志、Prometheus 指标和分布式 Trace。生产应监控 API P95、数据库连接池、Kafka lag、设备在线率、告警数量和外部 AI 成功率。

### 96. 后端应该重点测试哪些内容？

**参考回答：** 单元测试应覆盖状态机、角色权限、ETA、偏航距离、改址影响和 Agent 参数校验；集成测试应使用 Testcontainers 启动 PostgreSQL、Redis、Kafka，验证 Mapper、事务和消息链路；契约测试验证前后端响应字段。当前仓库没有形成完整自动化测试集，这是答辩时应如实说明的改进项。

### 97. 如何测试设备离线告警？

**参考回答：** 先发送正常心跳，确认 Redis TTL 为 90 秒且设备为 ONLINE；停止心跳，等待键过期和下一轮 30 秒检测，验证车辆状态改为 OFFLINE、只生成一条 DEVICE_OFFLINE 告警并 WebSocket 推送；再恢复心跳，验证设备回到 ONLINE。测试还要模拟 Redis 暂时故障，防止把基础设施故障误判成所有设备离线。

### 98. 生产环境如何部署？

**参考回答：** 前端静态资源由 Nginx 提供，`/api` 和 WebSocket 反向代理到 Spring Boot；后端 JAR 用 systemd 或容器运行；PostgreSQL、TimescaleDB、Redis、Kafka、EMQX 和 MinIO 独立部署并持久化。所有 API Key、数据库密码和 JWT Secret 通过环境变量或 Secret Manager 注入，不能继续硬编码在 YAML。

### 99. 当前项目最重要的安全改进是什么？

**参考回答：** 优先级依次是：把已暴露 API Key 全部轮换并移出 Git；将 MD5 密码迁移到 BCrypt/Argon2；实现货主订单数据范围隔离；收紧 CORS；给知识库管理和订单扩展接口补齐细粒度角色检查；限制登录和语音接口频率。安全不能只靠前端按钮隐藏。

### 100. 请用两分钟总结后端设计。

**参考回答：** 本项目后端使用 Spring Boot 2.7.3 和原生 MyBatis，主业务数据保存在 PostgreSQL，GPS 与设备心跳保存在 TimescaleDB，Redis承担最新位置、在线状态、会话和去重。车载终端通过 MQTT 接入，后端把 GPS 和心跳送入 Kafka异步持久化，再通过 WebSocket推送前端，形成设备到页面的实时闭环。业务层覆盖车辆、宠物运输记录、状态时间线、告警、调度指令、改址、司机评价和卸货核验，并通过事务保证主库内一致性。认证使用 JWT，后端执行角色校验；智能能力包括百度 ASR、人脸识别、DeepSeek、pgvector RAG、Agent 工具白名单和豆包 TTS。当前优势是链路完整、技术选择可解释，后续重点是密码算法、密钥管理、数据范围权限、幂等与自动化测试。

## 附录 B：答辩时必须主动区分的事实

- 项目使用原生 MyBatis，不是 MyBatis-Plus。

- 角色级权限已经实现，货主级订单数据隔离仍需完善。

- 豆包 TTS 代码已接入，但当前 Key 的资源授权仍需在供应商控制台开通。
- 实时链路已经包含 MQTT、Kafka、TimescaleDB、Redis 和 WebSocket，但完整事件幂等、死信队列和自动化测试仍是后续工作。
